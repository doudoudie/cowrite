# aodw写代码实践

通过aodw写代码的实践如下:

步骤一：描述需求
1、说出需求的输入、输出、关键逻辑 - 粗略的需求文档

步骤二、反思审查
2、用 AODW 的框架以"以终为始"的决策思路来反思审查需求文档 -> 输出的是待决策清单
3、澄清反馈的待决策清单
4、将澄清确认的问题，补充到需求文档

步骤三、决策
5、将需求文档执行生成RT结构目录 (决策文档、技术方案、影响分析、约束条件)

步骤四、审计
7、用 AODW 的框架 以"以终为始"的决策思路来审计需求文档 -> 输出一堆待决策清单和风险
8、澄清审计的问题清单
9、将澄清确认的问题，补充到需求文档
10、重复步骤 7 - 9；3-5轮的完善；
11、得到一份相对完整的需求文档

步骤五、执行
12、在 AODW 的框架下以"以终为始"的决策思路来写一份完整的测试用例，要求功能独立，单独可测(颗粒度尽量小)；
13、反过来通过测试用例验证需求文档的完整性
14、生产一份开发计划
15、开始执行这份开发计划
   我的开发计划 分3段，每一段又分第一点，第二点等等； 然后一点一点的执行
   一步完成后，除了要求他用测试用例测试外，还要问他下一步的任务是啥， 准备怎么做? 实现的目的是什么？
16、每一步执行完就让基于测试用例mork数据测试（测试用例有一些是基于边界以及测试数据的测试）

步骤六：开发审计、综合审计 (这两部还没执行、看了aodw的逻辑)

eg: 案例分析
1、有时候他会理解错需求，因为说得时候需求很大块，看着文字描述好像没错，但是在真正执行的时候，可能有偏差; (这里就是需要一步一步执行代码来及时纠正)
举例：模糊匹配
2、测试用例给我的启示： 并发问题、边界问题、安全问题;
3、做一个需求，你和ai沟通清楚要怎么做 分几步， 每一步的逻辑是啥 确认好后， 一步成功！
