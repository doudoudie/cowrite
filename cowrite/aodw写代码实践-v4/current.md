# 通过 AODW 写代码的实践

## 流程总览

描述需求 → 反思审查 → 决策 → 审计 → 执行 → 开发审计

## 步骤一：描述需求

1. 说出需求的输入、输出、关键逻辑，形成粗略的需求文档

## 步骤二：反思审查

1. 用 AODW 框架以"以终为始"的决策思路审查需求文档 → 输出待决策清单
2. 澄清反馈的待决策清单
3. 将澄清确认的问题补充到需求文档

## 步骤三：决策

### 核心思路

以终为始——从最终交付目标倒推，锁定关键技术路径和关键决策点，形成清晰的执行路线图。

### 1. 明确终态目标

1. 回顾需求文档中的"最终交付物"是什么
2. 用一句话描述："这个需求完成后的理想状态是……"
3. 明确验收标准：怎么算"完成"？谁说了算？

### 2. 关键路径决策

1. **技术选型**：用哪些技术/框架/库？为什么选这个而不是那个？
2. **架构方向**：单体还是分层？前后端分离还是耦合？数据怎么存储？
3. **依赖策略**：自研还是用现有轮子？第三方服务的可用性如何？
4. **风险路径**：哪个环节最可能出问题？有没有Plan B？

### 3. 生成 RT 结构目录

- **R - 决策文档（Resolution）**：关键决策点 + 决策理由 + 备选方案对比
- **T - 技术方案（Technical）**：详细技术实现路径、接口设计、数据结构、流程图

### 4. 决策记录格式

每个关键决策点记录：

```markdown
## 决策点：[具体问题]

- **选项A**：……
- **选项B**：……
- **选择**：A（选这个）
- **理由**：因为……
- **风险**：如果出错，Plan B 是……
```

### 5. 约束条件清单

1. **时间约束**：必须在什么时间点前完成？
2. **资源约束**：人力、设备、第三方服务是否充足？
3. **技术约束**：有哪些技术债务或历史包袱需要考虑？
4. **合规约束**：有没有安全、隐私、合规方面的要求？

### 6. 与 AI 对齐

1. 把 RT 文档交给 AI，让他复述理解
2. 确认关键路径理解一致
3. 有偏差立即纠正，不要等到开发中才发现

### 输出物

- RT 结构目录（已填写）
- 关键决策清单（无遗留未决项）
- 约束条件清单（时间/资源/技术/合规）

## 步骤四：审计

1. 用 AODW 框架以"以终为始"的决策思路审计需求文档 → 输出待决策清单和风险
2. 澄清审计的问题清单
3. 将澄清确认的问题补充到需求文档
4. 重复步骤 1-3，持续 3-5 轮完善
5. 得到一份相对完整的需求文档

## 步骤五：执行

### 核心思路

以终为始——把"最终交付"拆成一个个可测试的小目标，一步一步交付，每一步都用测试用例验证。

### 1. 写测试用例（TDD 前置）

1. **按功能点拆用例**：每个功能点独立一个测试用例
2. **用例要求**：
   - 功能独立：不依赖其他用例的执行结果
   - 单独可测：可以单独运行，不依赖上下文
   - 颗粒度小：一个用例只验证一个行为
3. **用例必须覆盖**：
   - 正常流程：输入正确时的预期输出
   - 边界条件：空值、极值、边界值
   - 并发场景：多线程/多用户同时操作
   - 安全场景：权限校验、SQL注入、XSS等

### 2. 用测试用例反查需求完整性

1. **对着测试用例过需求**：每个用例能否在需求文档里找到对应描述？
2. **找出缺失项**：有用例但需求没说清楚 → 补充需求
3. **找出冗余项**：需求写了但没有用例覆盖 → 确认是否真的需要
4. **结论**：需求文档 + 测试用例 → 两者一致才算完备

### 3. 生成开发计划

**格式**：分多段，每段分点

```
第一段：基础设施
  1.1 搭建项目框架
  1.2 配置CI/CD
  1.3 建立数据库表结构

第二段：核心功能
  2.1 实现XX模块
  2.2 实现YY模块
  2.3 前后端联调

第三段：收尾
  3.1 性能优化
  3.2 安全加固
  3.3 文档编写
```

### 4. 小步执行 + 实时验证

每一步执行的标准流程：

```
① 明确任务 → 这一步要做什么？产出是什么？
② 确认上下文 → 上一步的结果是否可用？有无前置依赖？
③ 执行实现 → 写代码
④ 用测试用例验证 → 用 mock 数据跑测试
⑤ 确认下一步 → 问AI：下一步是什么？准备怎么做？目的是？
⑥ 记录偏差 → 发现需求理解不一致 → 立即纠正
```

### 5. 测试数据策略

| 类型 | 举例 | 目的 |
|------|------|------|
| **正常数据** | 标准的、完整的输入 | 验证功能正确性 |
| **边界数据** | 空字符串、超长文本、特殊字符 | 验证边界处理 |
| **并发数据** | 多线程同时写入、重复提交 | 验证并发安全 |
| **异常数据** | 非法格式、权限不足、网络超时 | 验证容错能力 |
| **Mock数据** | 第三方接口Mock | 隔离依赖，稳定测试 |

### 6. 与 AI 协作节奏

```
用户/AI 每完成一步后：
  → 用测试用例验证
  → 问AI：下一步任务是什么？准备怎么做？目的是？
  → 确认理解一致再继续
  → 有偏差立即回滚或纠正
```

### 输出物

- 测试用例文档（功能独立、单独可测）
- 需求-测试用例对照表（无遗漏）
- 开发计划（分段时间，节点清晰）
- 每步的执行记录和验证结果

## 核心经验

- 需求要说清楚：输入、输出、每一步的逻辑，确认好再动手
- 小步执行：一步成功后及时纠正偏差，不要等到最后才发现需求理解错了
- 测试用例很重要：并发问题、边界问题、安全问题都能从测试用例中发现
- 以终为始：始终盯着最终目标思考每一步的对错
